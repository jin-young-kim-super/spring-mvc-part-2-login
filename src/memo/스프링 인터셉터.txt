서블릿 필터와 같이, 웹과 관련된 공통 관심(특정 HTTP 헤더,URL 요청에 대한 특정 로직 적용)을 처리하는 스프링이 제공하는 기술이 스프링 interceptor이다. 
그러나 적용되는 순서와 범위, 사용방법이 다르다.
※서블릿 필터보다 더 막강하다. 특별히 서블릿 필터를 꼭 사용해야 하는 상황이 아니라면 스프링 인터셉터를 사용하는 것이 더 편리함


호출 순서
HTTP 요청 -> WAS -> 【필터】-> (디스패처) 서블릿 -> 인터셉터 1 -> 인터셉터 2 ... -> 컨트롤러
-> 필터와 달리 서블릿과 컨트롤러 사이에서 호출된다.


스프링 인터셉터의 인터페이스
public interface HandlerInterceptor {
    default boolean preHandle(HttpServletRequest request, HttpServletResponse response, Object handler) throws Exception {}
    default void postHandle(HttpServletRequest request, HttpServletResponse response,Object handler, @Nullable ModelAndView modelAndView) throws Exception {}
    default void afterCompletion(HttpServletRequest request, HttpServletResponse response, Object handler, @Nullable Exception ex) throwsException {}
}

preHandle : 컨트롤러 호출 직전 호출
-> 더 정확히는 핸들러 어댑터 호출 직접에 호출
preHandle 반환값이 true 인 경우 : 핸드러 어댑터 호출
preHandle 반환값이 false인 경우 : 핸들러 어댑터 호출x

postHandle : 컨트롤러 실행 후 호출
-> 더 정확히는 핸들러 어댑터가 ModelAndView를 디스패처 서블릿에 반환한 직후에 postHandle 호출. 그래서 postHandle 호출 시점에는 어떤 Model과 View과 반환됐는지 볼 수가 있다.
그래서 postHandle의 매개변수에는 ModelAndView가 있다.

afterCompletion : 요청 완료 후 호출(이 시점에서는 postHandle이랑 헷갈릴 것이다)
-> 디스패처 서블릿에서 핸들러 어댑터가 반환한 ModelAndView를 가지고 View를 렌더링 완료하면 그떄 afterCompletion 호출


postHandle VS afterCompletion
이 둘의 차이점은 예외 발생 시에 드러난다.
만약에 컨트롤러 실행 중 예외가 발생을 한다고 하면, postHandle은 절대 호출되지 않는다. 그러나 afterCompletion은 예외가 발생하던 안 하던 무조건 호출된다.
그 이유는 이건 나의 추측이지만 afterCompletion의 매개변수에 @Nullable Exception ex가 있다.
즉 예외가 발생을 하지 않으면 ex == null이지만 예외가 발생을 하게 되면 이 매개변수에 어떤 예외인지에 대한 정보가 들어 있다.
afterCompletion은 3개의 메서드 중 가장 나중에 호출되는 것이기에 최대한 많은 예외 정보를 담을 수가 있다.
그래서 예외가 발생 시 postHandle은 호출이 되지 않아도 afterCompletion은 호출시켜서 개발자에서 어떤 예외가 발생했는지 알려 주지 위해서 호출하는 거 아닐까?
